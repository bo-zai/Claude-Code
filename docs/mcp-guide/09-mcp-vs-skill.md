# MCP 与 Skill 的区别

本文详细对比 MCP（Model Context Protocol）和 Skill（技能）两种扩展 AI 能力的方式。

---

## 一、概念定义

### MCP

**定义**：标准化的协议，用于连接 AI 客户端与外部工具/数据源。

```
MCP = 协议 + 连接标准

┌─────────────┐                ┌─────────────┐
│   AI 客户端  │ ←── JSON-RPC ──→ │  MCP Server │
│   (Agent)   │                │  (外部程序)  │
└─────────────┘                └─────────────┘
```

**本质**：一种通信协议，定义了如何让 AI 与外部程序交互。

### Skill

**定义**：预定义的能力模块，封装特定任务的专业知识和流程。

```
Skill = Prompt + 流程 + 专业知识

┌─────────────────────────────────────────┐
│                  Skill                   │
│                                         │
│  ┌─────────────┐  ┌─────────────┐       │
│  │  Prompt 模板 │  │  执行流程   │       │
│  └─────────────┘  └─────────────┘       │
│                                         │
│  ┌─────────────────────────────────┐   │
│  │         领域专业知识              │   │
│  └─────────────────────────────────┘   │
│                                         │
└─────────────────────────────────────────┘
```

**本质**：一种能力封装，定义了如何让 AI 以专业方式完成特定任务。

### Skill Script

Skill 支持在 Markdown 中嵌入 Shell 命令，实现本地脚本执行：

```markdown
---
shell: bash
---

请分析代码仓库，执行以下命令获取信息：

```! find . -name "*.ts" | head -20 ```

文件数量：!`find . -name "*.ts" | wc -l`
```

**执行方式**：
- `shell` frontmatter 指定默认 shell（bash 或 powershell）
- `!`command`` 内嵌命令
- ```! command ``` 代码块命令

## 二、核心对比

| 维度 | MCP | Skill |
|------|-----|-------|
| **定位** | 通信协议 | 能力模块 |
| **提供方** | 外部程序（Server） | Agent 内部定义 |
| **运行位置** | 独立进程 | Agent 内部 |
| **通信方式** | JSON-RPC | 直接函数调用 |
| **能力类型** | 工具执行、数据访问 | 任务流程、专业知识 |
| **可复用性** | 跨多个 AI 客户端 | Agent 特定 |
| **开发方式** | 编写独立程序 | 编写 Prompt + 配置 |
| **权限控制** | 需要用户审批 | 自动执行（通常） |

## 五、使用方式对比

### MCP 使用方式

```
用户: "查询代码仓库的结构"

Agent 处理:
1. 检测到需要外部工具
2. 从 MCP Server 获取工具定义
3. 将工具定义注入 Prompt
4. 发送给模型

模型响应:
{
  "tool_calls": [{
    "name": "query",              // MCP 工具名
    "arguments": { "query": "..." }
  }]
}

Agent 执行:
1. 检查权限
2. 发送 JSON-RPC 请求到 MCP Server
3. 接收结果
4. 反馈给模型

最终回答:
"这个代码仓库的结构如下..."
```

### Skill 使用方式

```
用户: "帮我审查这个代码"

用户触发 Skill:
用户输入: /code-review

或者用户描述任务:
"审查 src/auth.ts 的安全性"

Agent 处理:
1. 识别匹配的 Skill
2. 加载 Skill 的 Prompt 模板
3. 应用领域知识
4. 执行 Skill 定义的操作流程

Skill 执行流程（内部）:
┌─────────────────────────────────────────────────────┐
│              code-review Skill 执行                  │
├─────────────────────────────────────────────────────┤
│                                                     │
│  Step 1: 加载 Prompt 模板                           │
│  "请用 TypeScript 最佳实践审查代码..."               │
│                                                     │
│  Step 2: 应用领域知识                               │
│  注入安全规则、常见漏洞列表                          │
│                                                     │
│  Step 3: 调用 MCP 工具（可选）                       │
│  - read_file: 读取目标文件                          │
│  - query: 查询相关代码                              │
│                                                     │
│  Step 4: 生成结构化报告                             │
│  - 安全问题                                         │
│  - 性能问题                                         │
│  - 建议修复                                         │
│                                                     │
└─────────────────────────────────────────────────────┘

最终回答:
"代码审查报告：
  - 安全问题: 发现 3 处潜在漏洞
  - 建议修复: ..."
```

### 典型协作示例

```
用户: /code-review src/auth.ts

Skill 执行:

Step 1: Skill 解析参数
        - language: TypeScript
        - file_path: src/auth.ts

Step 2: Skill 调用 MCP 读取文件
        MCP call: read_file { path: "src/auth.ts" }
        MCP result: 文件内容

Step 3: Skill 调用 MCP 获取上下文
        MCP call: context { name: "login" }
        MCP result: 调用关系

Step 4: Skill 应用领域知识分析
        - 检查密码存储方式
        - 检查 SQL 注入风险
        - 检查权限控制

Step 5: Skill 生成报告
        输出结构化的审查报告
```

## 七、开发对比

### MCP 开发

```python
# MCP Server 开发
from mcp.server import Server
from mcp.types import Tool

app = Server("my-mcp-server")

# 1. 声明工具
@app.list_tools()
async def list_tools():
    return [
        Tool(
            name="query",
            description="查询代码知识图谱",
            inputSchema={
                "type": "object",
                "properties": {
                    "query": {"type": "string"}
                },
                "required": ["query"]
            }
        )
    ]

# 2. 实现工具逻辑
@app.call_tool()
async def call_tool(name: str, arguments: dict):
    if name == "query":
        # 执行查询操作
        result = await execute_query(arguments["query"])
        return result

# 3. 作为独立进程运行
if __name__ == "__main__":
    app.run(transport="stdio")
```

**特点**：
- 编写完整程序
- 定义工具接口
- 实现执行逻辑
- 作为独立进程运行
- 配置给 Agent 使用

### Skill 开发

```yaml
# Skill 配置文件
name: code-review
description: 代码审查技能
trigger:
  - "/code-review"
  - "审查代码"
  - "review this code"

prompt: |
  请用 {language} 的最佳实践审查以下代码：

  文件: {file_path}

  审查要点：
  1. 安全性检查
  2. 性能考虑
  3. 代码风格
  4. 可维护性

workflow:
  steps:
    - name: read_code
      action: read_file
      params:
        path: "{file_path}"

    - name: get_context
      action: context
      params:
        name: "{function_name}"
        optional: true

    - name: analyze
      action: model_analysis
      prompt: "{prompt}"

    - name: report
      action: generate_report
      format: structured

knowledge:
  - security_rules.yaml
  - best_practices/{language}.yaml
```

**特点**：
- 编写配置/模板
- 定义任务流程
- 注入领域知识
- 内置于 Agent
- 用户通过命令触发

## 十三、Skill Script vs MCP Tool

Skill 可以内嵌脚本执行能力（Script），这与 MCP Tool 都能执行操作，但本质上有明显区别。

### 本质区别

Skill Script 是 Skill **内部**的能力，脚本嵌入在 Skill 定义中，是 Skill 自带的一部分。MCP Tool 是**外部**通过协议提供的工具，工具定义在独立的 MCP Server 中。

```
┌─────────────────────────────────────────────────────────────────┐
│   Skill Script: 内部能力                                        │
│   ─────────────────────────────                                 │
│   • 脚本写在 Skill 文件/配置中                                   │
│   • 执行时机由 Skill 设计者决定                                  │
│   • 可预处理、运行时执行、或独立调用                              │
│                                                                 │
│   MCP Tool: 外部能力                                            │
│   ─────────────────────────────                                 │
│   • 工具在独立 MCP Server 中                                    │
│   • 执行时机由模型推理决定                                       │
│   • 通过 JSON-RPC 协议调用                                      │
└─────────────────────────────────────────────────────────────────┘
```

### 详细对比

| 维度 | Skill Script | MCP Tool |
|------|-------------|----------|
| **来源** | Skill 内部定义 | 外部 MCP Server |
| **执行时机** | 灵活，由 Skill 设计决定 | 模型推理阶段决策 |
| **执行方式** | Skill 内部直接执行 | JSON-RPC 跨进程调用 |
| **决策者** | Skill 设计者或用户 | 模型 |
| **能力范围** | Shell 命令、本地操作 | 任意复杂逻辑、外部服务 |
| **复用性** | 仅该 Skill 使用 | 调用该 Server 的所有场景 |
| **信任边界** | Skill 内部 | 外部 Server，需权限控制 |
| **开发方式** | 编写 Skill 时嵌入 | 开发独立 MCP Server |

### Skill Script 的三种执行时机

Skill 设计者可以灵活选择脚本执行时机：

**1. 预处理阶段**
- Skill 加载时执行
- 结果嵌入 Skill 的 Prompt
- 适用场景：收集静态信息、环境检测、项目结构分析

**2. 运行时阶段**
- Skill 运行过程中执行
- 类似 MCP Tool 的调用模式
- 适用场景：动态数据获取、状态检查、实时信息

**3. 独立执行**
- 用户单独调用脚本
- 不依赖 Skill 其他流程
- 适用场景：快捷命令、辅助工具、一次性操作

### MCP Tool 的执行流程

MCP Tool 的执行由模型驱动：

```
用户提问 → 构建 Prompt → 模型推理 → 决定调用 → 权限审批 → JSON-RPC 调用 → 返回结果 → 继续推理
```

整个过程中模型感知调用决策和结果，可以根据上下文决定是否调用、调用次数、调用顺序。

### 使用场景选择

**Skill Script 适用**：
- Skill 特定的本地操作（文件系统、Shell 命令）
- 执行时机需要 Skill 设计者控制
- 与 Skill 流程紧密配合，不需要复用

**MCP Tool 适用**：
- 需要模型动态决策是否调用
- 复杂逻辑（API、数据库、外部服务）
- 跨多个 Skill 或 Agent 复用
- 多轮交互、组合调用

### 协作示例

一个完整的 Skill 可以同时使用两种能力：

```
┌───────────────────────────────────────────────┐
│  Skill 执行流程                                │
│                                               │
│  1. Skill Script 预处理                       │
│     → 检测项目结构、收集文件信息                │
│                                               │
│  2. Skill 流程主体                            │
│     → 加载专业知识、构建分析框架                │
│                                               │
│  3. MCP Tool 调用                             │
│     → 查询代码知识图谱                         │
│     → 获取 GitHub PR 信息                     │
│                                               │
│  4. Skill Script 后处理                       │
│     → 格式化报告、整理输出                     │
└───────────────────────────────────────────────┘
```

Script 处理本地和简单操作，MCP Tool 处理外部和复杂操作，两者互补协作。

### 关键差异总结

**来源**：Script 内部定义，Tool 外部提供
**执行时机**：Script 灵活可控，Tool 模型决策
**决策权**：Script 属于设计者，Tool 属于模型
**能力范围**：Script 本地简单操作，Tool 复杂外部服务
**复用性**：Script 仅该 Skill，Tool 跨场景

一句话总结：Skill Script 是 Skill 的"自有工具"，MCP Tool 是"外部工具"。

## 总结

| 维度 | MCP | Skill |
|------|-----|-------|
| **本质** | 协议/接口 | 能力/流程 |
| **角色** | 工具执行者 | 任务编排者 |
| **位置** | 外部进程 | Agent 内部 |
| **关系** | 被调用 | 调用 MCP |

**一句话总结**：
- MCP 提供"手"（工具执行能力）
- Skill 提供"脑"（任务流程和专业知识）
- 两者协作：Skill 定义做什么，MCP 提供工具来做

**最佳实践**：复杂任务使用 Skill 编排 MCP 工具，简单操作直接使用 MCP 工具。
