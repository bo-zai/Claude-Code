  今天聊AJ记忆系统里一个大部分人没做好的环节，记忆的检索。上一期我们拆了cloud code的记忆分类体系，知道了什么该存，什么不该存，但光会存还不够，你存了200条记忆，用户问了一个问题，你是把200条全塞进上下文，还是只挑相关的几条全塞进去？Token成本爆炸，而且大部分内容跟当前对话无关，反而会干扰模型的推理，不塞进去记忆等于白存，这就是检索要解决的问题。Cloud code的源码里，这套检索机制叫active recall, 打开源码，你会发现它的设计比大多数人想象的要精巧的多。
不是简单的向量搜索，也不是关键词匹配，而是一个多阶段的筛选流水线，每一步都在控制成本和精度的平衡。先说整体流程，每一轮对话开始的时候，Active recall做四件事，扫描、摘要、选择、注入。我一步一步讲。第一步，扫描系统递归，扫描记忆目录，找到所有的记忆文件，上限是200个，超过200个的部分直接忽略，但它不读文件全文，它只读每个文件的前30行。
为什么是30行？因为Cloud code的记忆文件有一个固定格式，开头是一段叫front matter的原数据，包含记忆的名字，一句话描述类型。30行足够覆盖这段原数据了。这一步的成本极低，200个文件每个只读30行，总共也就几千行文本，没有任何模型调用。第二步，摘要扫描完之后，系统把所有记忆的原数据拼成一个清单，每条记忆一行，格式是类型、标签、文件名，最后修改时间，一句话描述。
比如反馈类型测试规范，三天前集成测试必须用真实数据库，不要mock，这个清单就是模型要看的目录，200条记忆的目录大概也就几千个头，肯比起把200个文件的全文都塞进去，成本降了几十倍。
第三步，选择。这是整个流程里最关键的一步。系统把用户当前的输入和记忆目录一起发给一个模型，让他从中选出最多5条跟当前对话最相关的记忆。注意，这里用的不是主模型，Cloud code里用的是sonnet，一个更便宜、更快的模型。选择记忆不需要最强的推理能力，只需要理解这条记忆跟当前问题有没有关系，Sonnet完全够用。选择模型的系统提示词也很讲究，它明确要求只选你确定有用的，不确定就不选，宁可少选，不要多选。
 如果没有任何记忆跟当前对话相关，返回空列表也完全可以。这个宁缺毋滥的策略很重要，因为每一条被选中的记都会占用主模型的上下文空间，选错一条就浪费了一个宝贵的位置，而且可能误导主模型的推理。还有一个细节，如果用户当前正在使用某些工具，而且用的很顺利，系统会告诉选择模型不要选这些工具的使用文档类记忆，因为模型已经在正常使用这些工具了，再三一份使用说明是噪声。但如果记忆里包含的是这些工具的已知问题或注意事项，那还是要选，因为正在使用的时候恰恰是最需要知道这些坑的时候。选择完成后，系统还有一个后校验步骤，模型返回的文件名，系统会逐一检查这些文件是否真的存在于候选集中，为什么？
  因为模型可能会幻觉出一个不存在的文件名，这一步是确定性代码兜底，防止模型的幻觉污染记忆检索结果。第4步，注入选中的记忆文件，系统读取他们的完整内容，然后作为系统提醒注入到当前对话的上下文中。但注入也不是无脑塞，每个文件有大小上限大约4000字节，超过的部分会被截段，但不是直接丢掉，而是附上一条提示，这个记忆文件被截断了。如果需要完整内容，可以用文件读取工具去读。这个设计和上下文工程那期讲的工具输出预算是同一个思路，不截断信息，而是给摘要加指针。每轮最多注入5条记忆，每条最多4000字节，所以单轮注入的上限大约是2万字节，整个会话的累计注入上限是6万字节。
  超过这个上限，Active recall就停止工作，不再注入新的记忆。为什么要有会话集的累计上限？因为长会话里如果每轮都注入5条记忆，几十轮下来光记忆就占了几十万字节的上下文累计上限，防止记忆把上下文撑爆。而且当会话触发上下文压缩的时候，之前注入的记忆会随着旧消息一起被压缩掉，累计计数器也会重置，Active recall又可以重新开始工作了。还有一个去重复机制，同一个会话里已经注入过的记忆文件不会被重复注入选择模型，在选择之前，系统就已经把已注入的文件从候选集里过滤掉了。
  这样选择模型的5个名额不会浪费在已经注入过的记忆上。4步讲完了，现在说一个贯穿整个流程的关键设计，过期警告。每条被注入的记忆都会附带一个时间标签，如果记忆是今天或昨天写的，不加任何警告，如果记忆超过2天了，系统会自动附加一段提醒，大意是这条记忆是多少天前的记忆是时间点快照，不是实时状态，里面关于代码行为或文件位置的描述可能已经过时了。在基于这条记忆做决策之前，请先验证当前代码的实际状态。
  源码注释里解释了为什么要这样做。有用户反馈说，旧的记忆里提到了某个函数在某个文件的某一行，模型就直接引用了这个信息，但实际上那个函数早就被重构到别的地方了。而且因为记忆里给出了具体的文件和行号，模型的回答听起来特别权威，用户一开始根本没怀疑过期警告就是为了打破这种虚假的权威感。
  告诉模型，这只是过去某个时间点的观察，不是当前事实。回过头来看整个active recall的设计，它本质上是一个两阶段检索系统。第一阶段用原数据做粗筛，成本极低，不调用模型，第二阶段用便宜模型做精选，成本可控，只有被选中的记忆才会被读取全文并注入上下文。这个架构把检索成本从所有记忆的全文大小降到了所有记忆的摘要大小，加上5条记忆的全文大小，200条记忆每条平均几千字节，全文加载要几十万字节。Active recall的方式，摘要加载只要几千字节，加上5条全文2万字节，总共不到3万字节。
  成本差了一个数量集。如果你在给自己的agent做记忆检索，这个两阶段架构可以直接借鉴最简单的实现方式，每条记忆文件的开头写一行描述。检索的时候，先把所有描述拼成一个列表，连同用户的问题一起发给一个便宜的模型，让他选出最相关的几条，然后只读这几条的全文，注入上下文。你不需要向量数据库，不需要embedding，不需要复杂的相似度计算。一个结构化的摘要加一次便宜的模型调用就够了。Cloud code用实际的生产数据证明了这一点。
  200条记忆，每轮只选5条，用sonnet做选择，成本和延迟都在可接受范围内。一句话总结，好的记忆检索，不是把所有记忆都给模型看，是让模型只看他此刻需要的那几条。先摘要后全文，用便宜模型做选择，给旧记忆加过期警告，这三个设计你今天就能用。
